# ソフトウェア開発201の鉄則 原理192:進化:言語は保守性に影響を与える


## 要旨

* ソフトウェア開発に使う「プログラミング言語」は、保守時の生産性に大きく影響する
* 機能開発は優れているが保守性はイマイチ、あるいはその逆の言語、あるいはどちらも優れている言語がある
* アセンブラなどの低水準な言語は、開発保守どちらに対しても不利だ

## 解説

この書籍が出た頃のプログラミング言語や開発環境は、おそらく、機能実現と保守性の両立が難しかった状況だったのだろう。
当時に比べたら劇的にコンピューティングパワーが増え、そして言語も高級なものが開発された現在、よほどのものでない限り、
保守性も機能実現も両方持っていルようだ。

ちょっと調べてみた。

| 言語 | 開発容易性 | 保守性 | 向いている用途 | 評価 |
|---|---|---|---|---|
| TypeScript | ◎ | ◎ | Web、Next.js、業務アプリ、API | 最有力 |
| Python | ◎ | △〜○ | AI、データ処理、自動化、試作 | 作りやすいが大規模化は注意 |
| Go | ○ | ◎ | API、CLI、バックエンド、インフラ系 | 長期保守に強い |
| Kotlin | ○ | ◎ | Android、Java資産活用、サーバー | Javaの現代版 |
| C# | ○ | ◎ | 業務システム、Windows、Unity、Web API | 企業向けに安定 |
| Rust | △ | ◎ | 高性能、組込み、CLI、インフラ | 難しいが品質は高い |
| Java | △〜○ | ◎ | 大規模業務、既存システム | 堅実だがやや重い |
| JavaScript | ◎ | △ | 小規模Web、スクリプト | TypeScript推奨 |
| Swift | ○ | ○ | iOS/macOS | Apple専用なら良い |
| Dart | ○ | ○ | Flutterアプリ | モバイルUI特化 |

今の時代、大事なのは、「開発対象に適しているか」という点だろうか。
高級言語が使えるのに低水準なものを使うとか、高速化が必要な局面で高級すぎて性能が出せないものを使うとか、
そういうミスマッチさえなければ、大丈夫だろう。

ただ、言語選択は「何を開発するか」で行うのが常だが、将来的な保守まで考えて行えると、なお、いい。
